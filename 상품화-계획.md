# webpilot 상품화 계획 (사내 100명 배포)

## 아키텍처 결정 (확정)
- **모델 A — 로컬 데스크톱 앱**. 자동화는 각자 PC에서 실행. 서버는 배포·업데이트만.
- 셸: **Tauri v2** (`.dmg` / `.msi`, 아이콘, 자체 창).
- 백엔드: 기존 `server.mjs` + Playwright 그대로 **node 사이드카**로 동봉 (재작성 0).
- 브라우저: Playwright `channel:'chrome'` (시스템 크롬 재활용, 크로미움 번들 X).
- 자격증명: **각자 PC 로컬 암호화 볼트에만**. 서버 절대 전송 X. (모델 B 중앙실행은 비번 집중·부하 폭탄이라 폐기)
- 업데이트: Tauri updater가 **내 서버**의 `latest.json` 폴링 → 마크업 깨지면 고쳐서 푸시.

## 먼저 결정할 것 (착수 전 블로커)

### 1. 코드 서명 (★가장 큰 비용 결정)
서명 안 하면 100명이 무서운 경고 만남 = 문의 100건.
- **맥**: Apple Developer Program ($99/년) → notarization. 없으면 "확인되지 않은 개발자" → 우클릭 열기 안내 × 100.
- **윈도우**: 코드사인 인증서(연 수만~수십만원) 없으면 SmartScreen "PC 보호" 경고.
- 결정: 서명 살까? 아니면 첫 배포는 무서명 + 사내 안내로 감수?

### 2. 빌드 머신 (크로스빌드 불가)
- 맥 빌드 = 맥, 윈도우 빌드 = 윈도우 필수.
- 옵션 A: GitHub Actions 매트릭스(mac+windows 러너 무료) → 아티팩트 내 서버 업로드.
- 옵션 B: 맥 로컬 + 윈도우 머신/VM 수동.
- 결정: 윈도우 머신 접근 되나? CI 쓸까?

### 3. 배포 서버 스펙
- 정적 파일 서빙만 되면 됨 (`.dmg`/`.msi` + `latest.json` + 서명 아티팩트).
- HTTPS 필수 (updater가 http 거부).

### 4. 사내 IT/보안 사전 브리핑
- "비번 로컬 전용, 127.0.0.1, 외부 전송 0" 설명 문서 1장. 100명 배포 전 승인받아 두면 안전.
- (참고) 타임인아웃·비즈플레이·flow.team 약관의 자동화 조항 = 사내용이라 리스크 낮지만 확인해 두기.

### 5. 맥/윈도우 비율
- 100명 중 비율 → 어느 산출물 우선. 둘 다면 둘 다 빌드.

## 구현 단계

### Phase 1 — 서버 패키징 대응 (코드 리팩터)
패키징되면 앱 리소스는 **읽기전용**. 지금 CWD 기준 경로 전부 깨짐. 사용자 쓰기 가능 폴더로 이전.
- `.auth/accounts.enc`, `snapshots/`, `screenshots/`, 다운로드 엑셀 → **앱데이터 폴더**로.
  - 맥: `~/Library/Application Support/webpilot/`
  - 윈도우: `%APPDATA%\webpilot\`
- 방법: Tauri가 사이드카 스폰 시 `WEBPILOT_DATA_DIR` 환경변수 주입 → `server.mjs`가 그 밑에 읽기/쓰기.
- `express.static(join(__dirname,'public'))` 는 이미 처리됨(b1e4893). `src/`도 동봉 리소스로.
- 포트: 고정 8181 충돌 위험 → 빈 포트 자동 탐색 후 Tauri에 전달.

### Phase 2 — Tauri 셸 + 사이드카 배선
- `create-tauri-app` (v2), vanilla.
- 앱 시작 → node 사이드카 스폰(`node resources/server.mjs`) → 포트 열릴 때까지 폴링 → 창을 `http://127.0.0.1:PORT` 로 이동.
- 부팅 중 스플래시 표시.
- 앱 종료 → 사이드카 kill (좀비 방지).

### Phase 3 — node + node_modules 동봉 (OS별)
- 공식 node 바이너리를 타깃 트리플 이름으로 사이드카 등록:
  - `node-x86_64-apple-darwin`, `node-aarch64-apple-darwin`, `node-x86_64-pc-windows-msvc.exe`
- `server.mjs` + `src/` + `public/` + `node_modules/` = Tauri `resources`.
- node_modules는 **OS별로** 준비 (Playwright 드라이버가 플랫폼 종속). `channel:'chrome'`라 크로미움은 제외 → 가벼움.
- 크롬 미설치자 대비: 최초 실행 시 크롬 감지 → 없으면 설치 안내(또는 크로미움 폴백).

### Phase 4 — 자동 업데이트 + 서명
- Tauri updater 플러그인. `tauri.conf.json` endpoints = `https://내서버/updates/{{target}}/{{arch}}/{{current_version}}` (또는 정적 `latest.json`).
- updater 서명 키쌍 생성 → 공개키 config, 개인키로 빌드 시 아티팩트 서명.
- 서버가 서빙: `latest.json`(버전·릴리스노트·플랫폼별 URL+서명) + 업뎃 번들.
- 배포 흐름: 버전 태그 → 빌드 → 서버 업로드 → 앱들이 자동 감지·설치.

### Phase 5 — 빌드 파이프라인
- GitHub Actions 매트릭스(macos-latest, windows-latest):
  - node_modules 설치(OS별) → node 바이너리 fetch → tauri build → 서명 → 내 서버로 업로드.
- 버전 = `package.json` + git 태그 단일 소스.

### Phase 6 — 배포 & 롤아웃
- 파일럿 5명 → 피드백 → 100명 확대.
- 설치 안내 1장(맥/윈도우, 서명 경고 대처, 크롬 필요).
- 문의 창구 + 롤백 절차(서버에 이전 버전 유지).

## 리스크 & 대응 (하중 순)
1. **마크업 깨짐** — 세 서비스가 UI 바꾸면 100명 동시 마비. → Phase 4 자동업뎃이 생명줄. 정규식 셀렉터 방어적으로, 깨짐 감지 시 사용자에 명확한 에러.
2. **코드 서명 비용/경고** — 무서명 = 배포 마찰. 서명 결정 선행.
3. **크롬 의존** — `channel:'chrome'`는 크롬 있어야. 미설치 폴백 or 안내 필요.
4. **자격증명 보안** — 로컬 전용 사수. 앱데이터 볼트 권한/암호화 점검. 서버로 새는 코드 0 유지.
5. **크로스빌드** — CI 매트릭스로 해소.

## 안 하는 것 (폐기 확정)
- pkg/SEA 단일 바이너리 (ESM+Playwright로 막힘)
- 중앙서버 실행(모델 B) — 비번 집중·부하
- Electron (Tauri 대비 +100MB, 이득 없음)
- Playwright MCP / LLM — 이건 AI 제품 아님, 결정론 로직이라 부적합
- 크롬 확장 — 전체 재작성 + 백그라운드 상실
